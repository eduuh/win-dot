# Architecture

Why win-dot is shaped the way it is.

---

## Bare-repo dotfiles

Dotfiles live in a **bare** git repo at `~/projects/win-dot-bare`, with the *work tree* pointing at `$HOME`. That means every tracked file lands at the exact path it's used (`~/.tmux.conf`, `~/.config/powershell/profile.ps1`, etc.) without symlinks, copy steps, or template engines.

A `dot` shell function does

```powershell
git --git-dir="$HOME/projects/win-dot-bare" --work-tree="$HOME" @args
```

so it behaves like `git` but operates on the bare repo. `dot status`, `dot add ~/.config/...`, `dot commit`, `dot push` — same muscle memory.

A plain clone at `~/projects/win-dot` exists alongside the bare repo for browsing, editing scripts, and running tools that expect a normal `.git/` directory. The two are independent clones; both push to `origin`.

### Why bare?

| Alternative | Trade-off |
|---|---|
| Symlinks from a single dotfiles dir | Breaks when target paths are inside read-only or virtualised locations (e.g. OneDrive-redirected `Documents/`). Doubles the file count in your home dir. |
| Stow / chezmoi / yadm | Adds a runtime dependency; templating creates a second source of truth. |
| **Bare repo (this approach)** | Zero dependencies beyond git. The repo *is* the home dir layout. |

The pattern is the same on the WSL side (see `~/projects/dotfiles`).

## Profile stubs

PowerShell's `$PROFILE` path is not portable across machines:

| Host | Path (with OneDrive KFM active) | Path (without KFM) |
|---|---|---|
| pwsh 7 | `~/OneDrive - <Tenant>/Documents/PowerShell/Microsoft.PowerShell_profile.ps1` | `~/Documents/PowerShell/Microsoft.PowerShell_profile.ps1` |
| Windows PowerShell 5.1 | `~/OneDrive - <Tenant>/Documents/WindowsPowerShell/Microsoft.PowerShell_profile.ps1` | `~/Documents/WindowsPowerShell/Microsoft.PowerShell_profile.ps1` |

Tracking any of these is non-portable: the tenant name is org-specific, KFM status varies, and pwsh 7 + 5.1 have different paths. **So we don't.**

Instead, the repo tracks exactly one canonical file:

```
~/.config/powershell/profile.ps1
```

…and `scripts/install-packages.ps1` (function `Install-ProfileStubs`) writes a tiny dot-source stub at whatever `$PROFILE` resolves to on the current machine, for both pwsh 7 and Windows PowerShell 5.1:

```powershell
# Auto-generated by scripts/install-packages.ps1
$localProfile = Join-Path $HOME '.config\powershell\profile.ps1'
if (Test-Path $localProfile) { . $localProfile }
```

The stub installer is **idempotent** (skips if a `~/.config/powershell/profile.ps1` reference is already present) and **preserves user content** (existing profile content is kept; the stub is appended).

The path `~/.config/powershell/` is XDG-Base-Directory-style but works fine on Windows — it's just a folder name, no OS magic.

## Private overrides

Some things should never be committed to a public repo:
- Work email, tenant IDs, Azure DevOps credential helpers
- Machine-specific helper functions

Two off-ramps are provided, both gitignored:

### `~/.gitconfig.local`

The tracked `~/.gitconfig` ends with:

```ini
[include]
    path = ~/.gitconfig.local
```

git merges this on `--list --includes` and at credential-helper resolution time. See `.gitconfig.local.example` for the shape.

### `~/.config/powershell/profile.local.ps1`

The tracked `~/.config/powershell/profile.ps1` ends with:

```powershell
$localProfile = Join-Path $HOME '.config\powershell\profile.local.ps1'
if (Test-Path $localProfile) { . $localProfile }
```

So you can drop identity-switcher functions, work-only aliases, or secrets-loading code there. See `.config/powershell/profile.local.ps1.example` for the shape.

### What protects you

`~/projects/win-dot/.gitignore` (also visible to the bare repo via `dot`):

```
.gitconfig.local
.config/powershell/profile.local.ps1
.claude/
```

A `dot add` of either of those files is **refused**, not silently accepted with a warning.

## psmux session model

The Windows Terminal `powershell-tmux` profile launches `~/.config/tmux/start-main.ps1`, which:

1. Calls `psmux has-session -t main`.
2. If the session doesn't exist, creates it with:
   - Window 1 (`notes`): `hx ~/projects/branch-notes-work` if Helix is installed and the dir exists, otherwise a plain shell.
   - Window 2 (`htop`): `btop` if installed.
3. `psmux attach -t main`.

That makes re-opening Terminal a free no-op (just reattach), and a fresh boot get a sensible default layout. The script is idempotent and graceful: missing tools degrade to a plain shell window, never an error popup.

## Helix config

The editor on Windows is Helix (`hx`). Config lives at `AppData/Roaming/helix/` — that's where `hx --health` shows the config path on Windows, so we track it there rather than under `.config/helix/`.

Two files:

- **`config.toml`** — theme (`modus_vivendi` to match the nvim setup), editor options ported from `~/.config/nvim/lua/config/options.lua` (scrolloff, soft-wrap, indent guides, cursorline, etc.), and a small set of key overrides that preserve nvim muscle memory:
  - `C-s` save (normal + insert)
  - `backspace` → previous buffer (nvim `<BS> = <C-^>`)
  - `;n` / `;p` → next / previous buffer (nvim `cnext` / `cprev`)
  - `<space>ff` / `<space>fw` / `<space>fo` / `<space>fb` — picker aliases matching `<leader>f*` in nvim
  - `gcc` → toggle line comment (Helix native is `C`, but `gcc` is cheap to add)

  Everything else stays Helix-default — `gd / gr / gi / gy / K / [d / ]d / <space>r / <space>a / <space>s / <space>S / <space>b / <space>f / <space>/` already match the equivalent nvim bindings.

- **`languages.toml`** — language servers (rust-analyzer, typescript-language-server, clangd, lua-language-server, marksman, bash-language-server, pylsp, vscode-css-language-server) and per-filetype formatters (rustfmt, prettier, clang-format, stylua, shfmt, black, d2 fmt). Auto-format on save mirrors `conform.nvim`. Helix has no Mason, so tools must be on PATH — `scripts/install-packages.ps1` adds the necessary scoop / npm / pip installs.

The Helix runtime directory (`AppData/Roaming/helix/runtime/`) is gitignored — it's bundled with the binary and version-specific.

## Windows Terminal profiles

Only two profiles are visible — the rest are auto-discovered by Terminal and hidden via `"hidden": true`:

- **`powershell-tmux`** (default) → `pwsh -NoLogo -NoExit -File "%USERPROFILE%\.config\tmux\start-main.ps1"`
- **`ubuntu-wsl`** → the WSL 2 distro

`pwsh` is invoked via **absolute path** (`C:\Program Files\PowerShell\7-preview\pwsh.exe`) to dodge a known bug in the `Microsoft.PowerShell.Preview` winget package: it appends a non-existent `...\7-preview\preview` subdir to the Machine PATH, so plain `pwsh.exe` lookup fails with `0x80070002`. The installer also fixes the User PATH defensively.

---

## Why `Microsoft.PowerShell.Preview`?

It's not strictly required; you could use the stable `Microsoft.PowerShell` package. Preview is pinned so that new pwsh features land here first. If you'd rather track stable, swap the ID in `scripts/install-packages.ps1` and remove the `7-preview` PATH-fix block.

## Verification

`scripts/verify-environment.ps1` is a 25-check smoke suite. Run it after `install-packages.ps1` to catch drift:

```powershell
pwsh -NoProfile -File .\scripts\verify-environment.ps1
```

It exercises:
- Tools resolve on PATH (git, gh, pwsh, hx, d2, btop, psmux, …)
- `~/.tmux.conf` parses without error in psmux
- Windows Terminal `settings.json` has the expected profile shape
- `d2w` starts and serves a watch page on localhost
- `install-packages.ps1` parses (AST check)
- Tracked-vs-live file hashes match for files that are mirrored
